# Epoch 중간 끊김 안전성 분석

## 1. Epoch 중간에 끊어도 괜찮은 이유

### 딥러닝 학습의 실제 동작

```
epoch 1: [step1] → [step2] → ... → [step500] → epoch 완료
                                 ↑
                           여기서 끊어도?
```

각 step은 **독립적인 gradient update**입니다. step 300에서 끊어도:
- step 1~300의 학습은 **이미 모델 가중치에 반영**되어 있음
- 날아가는 건 step 301~500의 **"아직 안 한" 학습**뿐
- 이미 본 데이터에 대한 학습은 **되돌릴 수 없이 완료**된 상태

> 비유: 책 10장 중 7장까지 읽다가 멈춘 것이지, 7장까지 읽은 기억이 사라지는 게 아닙니다.

### Autoresearch가 안전한 이유

1. 5분 동안 가능한 만큼 step을 돈다
2. **끊은 후 평가(eval)를 실행한다** ← 이게 핵심
3. 평가 결과로 판단한다

"3.7 epoch 모델" vs "4.2 epoch 모델"이 아니라, **둘 다 정확히 5분 학습한 모델의 성능**을 비교하는 것입니다.

## 2. 걱정 안 해도 되는 것 vs 진짜 주의할 점

### ✅ 걱정 안 해도 되는 것

| 상황 | 왜 괜찮은가 |
|------|-----------|
| 3.5 epoch에서 끊김 | 3.5 epoch분의 학습은 완전히 반영됨 |
| 데이터 일부만 봄 | 매 epoch마다 shuffle → 편향 없음 |
| 수렴 안 됨 | **비교 대상도 수렴 안 됨** → 공정한 비교 |

### ⚠️ 실제로 주의할 점

#### 1. 평가를 반드시 학습 후에 해야 함

```python
# ❌ 위험: epoch 끝에만 평가하는 현재 코드
for epoch in range(epochs):
    train_one_epoch(...)
    if (epoch + 1) % eval_every == 0:
        evaluate(...)  # 중간에 끊기면 이걸 안 탐!

# ✅ 안전: 시간 초과 후 반드시 평가
while time.time() - start < TIME_BUDGET:
    train_one_step(...)
# 루프 탈출 후 무조건 평가
metrics = evaluate(model, val_loader, device)  # 반드시 실행
print(f"val_mean_r10: {metrics['mean_r10']:.6f}")
```

#### 2. Learning Rate Warmup 구간에서 끊기면 불공정

```
LR: 0 ──────▶ 최대 ──────────────▶ 0
     [warmup]     [학습]        [decay]
     
만약 5분이 warmup 안에서 끝나면?
→ LR이 아직 올라가는 중이라 "제대로 학습 시작도 안 한" 상태
```

**해결**: warmup을 시간 비율로 설정
```python
# 현재: warmup_epochs=3.0 (epoch 기반)
# 변경: warmup을 전체 시간의 10%로 고정
warmup_ratio = 0.1  # 5분 중 30초만 warmup
```

#### 3. BatchNorm 통계

MobileNetV4에는 BatchNorm이 있어 학습 데이터의 통계(mean/var)를 추적합니다. 너무 짧으면 통계가 불안정할 수 있지만, **5분이면 수백~수천 step**이므로 실질적 문제 없음.

## 3. dual-distillation_ver2에서 권장하는 방식

위 위험성을 고려하면, **축소 데이터 + 짧은 epoch 완주**가 시간 예산 방식보다 더 안전합니다:

```
시간 예산 방식:    5분 → epoch 중간에 끊길 수 있음 → 평가 누락 위험
축소 데이터 방식:  10% 데이터 × 3 epoch → 자연스럽게 완주 → 정상 평가 ✅
```

| 방식                | 중간 끊김 위험 | 평가 누락    | 추천도   |
| ----------------- | -------- | -------- | ----- |
| 시간 예산 (5분)        | 있음       | 코드 수정 필요 | ⭐⭐⭐   |
| 10% 데이터 + 3 epoch | 없음       | 정상 작동    | ⭐⭐⭐⭐⭐ |
| 10% 데이터 + 시간 예산   | 있음       | 코드 수정 필요 | ⭐⭐⭐⭐  |

> [!IMPORTANT]
> **결론**: 현재 코드 구조에서는 **데이터를 줄이고 epoch을 완주**시키는 것이 코드 수정도 최소화하면서 가장 안전합니다. 시간 예산 방식은 `train.py`에 "루프 탈출 후 반드시 평가" 로직을 추가해야만 안전합니다.

## 4. 데이터 축소 시 발생할 수 있는 문제점

### 핵심 문제: "10%에서 좋았는데 100%에서 안 좋은" 순위 역전

```
10% 데이터 실험 결과:
  설정 A: R@10 = 0.42  ← 1등 🥇
  설정 B: R@10 = 0.40  ← 2등

100% 데이터 실험 결과:
  설정 A: R@10 = 0.55  ← 2등 😱
  설정 B: R@10 = 0.57  ← 1등 🥇  ← 역전!
```

**왜 발생하는가:**
- 특정 하이퍼파라미터(weight decay, LR schedule 등)는 **데이터 양에 따라 최적값이 달라짐**
- 적은 데이터 → 과적합 쉬움 → regularization이 강한 설정이 유리
- 많은 데이터 → 과적합 걱정 ↓ → 공격적인 설정이 유리

### 데이터 다양성 손실

LPCVC 데이터는 여러 소스(coco, flickr30k, open_images, wit) 포함:

```
전체 데이터:  COCO 60% + Flickr 15% + OpenImages 15% + WIT 10%
10% 샘플링:   비율은 유지되지만...
              → 드문 카테고리의 이미지가 아예 빠질 수 있음
              → "희귀 케이스에 강한 설정"을 놓칠 수 있음
```

### Batch 내 Negative 품질 저하

Contrastive learning(SigLIP, Hard Negative 등)은 **배치 안의 negative 다양성**이 중요합니다:

```
전체 데이터 + batch_size=128:
  → 다양한 이미지/텍스트 → 좋은 negative

10% 데이터 + batch_size=128:
  → 데이터풀이 작아 비슷한 샘플 반복 → negative 품질 ↓
  → text_text_contrastive_loss에 특히 영향
```

### 실험 유형별 신뢰도

| 실험 유형 | 10%에서 신뢰 가능? | 이유 |
|----------|------------------|------|
| LR 크기 비교 | ✅ 대부분 유효 | 수렴 방향은 데이터 양과 무관 |
| Loss 함수 종류 변경 | ✅ 대부분 유효 | 기본 트렌드는 유지됨 |
| Loss 가중치 미세 조정 | ⚠️ 순위 역전 가능 | 최적 비율이 데이터 양에 따라 변함 |
| embed_dim 변경 | ✅ 유효 | 구조적 차이는 데이터 양과 무관 |
| Distill temperature | ⚠️ 주의 | Teacher-Student gap이 달라짐 |
| 모델 아키텍처 변경 | ✅ 대부분 유효 | 큰 구조 차이는 작은 데이터에서도 드러남 |
| Regularization 강도 | ❌ 신뢰 어려움 | 과적합 정도가 데이터 양에 직접 연동 |

### 대처법

> **"10%로 방향을 잡고, 100%로 확인한다"**

10% 실험에서 **확실히 나쁜 것을 걸러내는 용도**로 쓰고, 상위 후보는 반드시 100%로 재검증합니다. 10%에서 나쁜 건 100%에서도 **거의 항상 나쁩니다** — 이 방향의 신뢰도는 높습니다.


### 궁금증

'시간 예산 방식에서 항상 다른 데이터셋이 들어가도록 학습'
-> 이것은 학습 데이터 셋을 로드시 랜덤으로 들어가도록 되어있는 거야? 
-> 또 그렇게 되면 각 설정 테스트마다 데이터셋의 구성에 따라 다른 정확도가 나오는 문제가 없어?

1. 데이터를 뽑는 방식
	현재 우리코드는 **shuffle** 로 데이터셋을 불러오도록 설정 되어있습니다.
	이는 어떤 데이터셋 (카드팩 A)를 뽑고
	처음에 한번 섞은 후 이를 각 배치마다 동일한 n개 만큼 뽑아서 모델에게 주는 이런 방식임
	때문에 어차피 항상 "랜덤"의 데이터가 제공된다.

2. 시간 예산 방식으로 각 학습마다 데이터의 구성, 데이터의 제공 순서가 바뀌어 서로 제공되는 데이터에 따른 정확도나 학습도의 차이가 있지않느냐? 
	이에 대한 답은 **Seed** 라는 것이 설정되어있는데 이것으로 우리는 그 카드가 섞인 순서를 고정 시킬 수 있습니다 때문에 제공되는 데이터는 각 학습마다 동일합니다.
	결과적으로 제공되는 데이터에 따른 운(노이즈)은 배제되며, 
	**오직 우리가 변경한 모델 설정의 차이만 정확도에 반영**되므로 신뢰할 수 있는 비교가 가능합니다."


### 우리는 시간 예산제가 아닌 데이터셋을 줄이며 적은 Epoch 으로 이 Auto Research를 사용 하는 방향이 맞아 보이는 이유
### 카파시가 남긴 조언 (README 발췌)

카파시는 "만약 5분이라는 짧은 시간(혹은 작은 컴퓨터)으로 유의미한 결과를 내고 싶다면 이렇게 세팅을 바꿔라"라며 7가지 조언을 했는데요, 그중 1번이 바로 이것입니다.

> **1. "To get half-decent results I'd use a dataset with a lot less entropy, e.g. this TinyStories dataset. ... Because the data is a lot narrower in scope, you will see reasonable results with a lot smaller models."** (그럴싸한 결과를 얻으려면 **복잡도(Entropy)가 훨씬 적은 데이터셋을 사용하라.** 범위가 좁은 데이터를 써야만 작은 모델과 짧은 시간으로도 '합리적인(납득할 만한) 결과'를 볼 수 있다.)

### 비전(Vision) 분야의 언어로 번역하면?

카파시가 말한 "크고 복잡한 위키피디아 데이터를 5분 만에 겉핥기 식으로 보지 말고, **작고 좁은 데이터셋(TinyStories)을 줘서 그 안에서 제대로 된 패턴을 학습하게 만들어라**"라는 철학은,

여러분의 컴퓨터 비전/멀티모달 프로젝트 상황에 대입하면 완벽하게 **"50만 장을 5분 동안 0.16 Epoch 훑어보지 말고, 데이터셋을 10%(5만 장)로 줄이되 3 Epoch을 제대로 돌려라"** 라는 결론과 100% 일치합니다.

즉, 카파시 역시 **"주어진 시간(예산) 내에 모델이 '실제로 수렴(Converge)'할 수 있는 수준으로 데이터의 판을 작게 줄여주는 것"** 이 빠른 탐색 실험에서 가장 중요하다는 것을 정확히 짚고 넘어간 것입니다!
